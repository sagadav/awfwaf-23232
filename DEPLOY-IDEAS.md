# Идеи на будущее (деплой)

Черновик улучшений процесса деплоя frontend1 на AlwaysData — не обязательно делать сразу. Актуальная инструкция: [DEPLOY.md](./DEPLOY.md).

- **Автоматизация**: один скрипт (`deploy.ps1` / `npm run deploy`): сборка → заливка → при возможности перезапуск без панели (SSH, API AlwaysData или их механизм перезапуска — уточнить у провайдера).
- **Заливка**: `rsync` вместо полного `scp -r` для инкрементальных деплоев (на Windows — WSL или отдельный `rsync`).
- **Проверка после деплоя**: запрос к прод-URL, опционально `/health` или отображение версии/хеша билда в ответе.
- **Откат**: куда хранить предыдущий `.output` и как быстро вернуться на него.
- **Конфиг продакшена**: где задаются `NUXT_*` / runtime config на AlwaysData; как менять без лишней пересборки, если применимо.
- **Гейт перед выкладкой**: линт/тесты до заливки.
- **CI**: сборка и деплой из GitHub Actions (или аналога) по тегу/ручному запуску.
- **Документация**: зафиксировать в [DEPLOY.md](./DEPLOY.md) точные настройки панели (рабочий каталог, команда, порт, переменные) после финальной настройки сайта.
