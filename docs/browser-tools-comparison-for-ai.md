# Сравнение способов автоматизации браузера для ИИ-агента

**Контекст:** как ассистенту в Cursor удобнее всего взаимодействовать с браузером — через **vibe-tools browser** (Stagehand), **browser-use**, или **Playwright MCP**.

**Дата:** 2026-04-12  
**Проект:** в `vibe-tools.config.json` уже настроены `stagehand` (Anthropic) и `mcp` (Anthropic + модель для MCP).

---

## 1. Краткая сводка

| Критерий | vibe-tools browser (Stagehand) | browser-use | Playwright MCP |
|----------|-------------------------------|-------------|----------------|
| **Где живёт логика** | CLI, вызывается из терминала агентом | Отдельный Python-стек / свой рантайм | MCP-сервер + клиент (Cursor) |
| **Интеграция с Cursor** | Через shell — надёжно, без отдельного MCP | Нужен запуск процесса / API | Нативные инструменты MCP в чате |
| **Модель «естественного языка»** | Да (`act`, `observe`, `extract`) | Да, ядро — агент поверх браузера | Зависит от сервера: часто низкоуровневые шаги |
| **Сессия и состояние** | По умолчанию stateless; есть `--connect-to` | Сессии в своём процессе | Зависит от реализации сервера |
| **Зависимости** | Playwright под капотом; может понадобиться установка | Python, браузер, зависимости пакета | Node/Python + MCP-сервер + конфиг Cursor |

**Практический вывод для агента в этом репозитории:** первым инструментом разумно использовать **vibe-tools browser**, т.к. он уже описан в правилах workspace, настроен в `vibe-tools.config.json`, и даёт высокоуровневые команды из одной оболочки. **Playwright MCP** — когда нужны частые итерации «клик/селектор» прямо из диалога без CLI. **browser-use** — когда автоматизация должна жить в Python-пайплайне вне Cursor или нужен долгоживущий агентный цикл с собственной оркестрацией.

---

## 2. vibe-tools browser (Stagehand)

**Что это:** обёртка над Stagehand — NL-команды (`open`, `act`, `observe`, `extract`), опционально скриншоты, сеть, консоль.

**Плюсы для ИИ:**

- Один вызов из терминала с полной инструкцией на человеческом языке (в т.ч. цепочки через `|`).
- Захват HTML, сети, консоли — удобно для отладки фронтенда.
- В проекте уже заданы провайдер/модель для Stagehand.
- Не требует отдельной регистрации MCP в Cursor.

**Минусы:**

- Каждый запуск — отдельный процесс (если не использовать `--connect-to` к уже запущенному Chrome).
- Ограничения из правил (например, отключённый `wait` внутри `act`) нужно помнить при сценариях.
- Зависимость от глобального `vibe-tools` и ключей в `~/.vibe-tools/.env`.

**Когда выбирать:** E2E-smoke, «открой страницу и проверь», извлечение данных с сайта, сценарии в стиле «сделай несколько шагов одной строкой», отладка сети/консоли.

---

## 3. browser-use

**Что это:** экосистема (часто Python) для LLM-агентов, управляющих браузером: планирование, память, инструменты — ориентирована на автономных агентов, а не на одноразовый вызов из IDE.

**Плюсы:**

- Сильная модель «агент думает → действует в браузере» в длительных задачах.
- Удобно встраивать в бэкенд, cron, отдельные сервисы.

**Минусы для агента в Cursor:**

- Нет прямого «родного» канала из чата Cursor: нужен отдельный процесс, зависимости, возможно API.
- Дублирует по смыслу то, что даёт Stagehand + LLM, но с другим деплоем и стоимостью сопровождения.
- Для разовых проверок UI в репозитории — избыточно.

**Когда выбирать:** автономные боты, офлайн-пайплайны, когда браузерная автоматизация — часть Python-продукта, а не задача «помочь разработчику в IDE».

---

## 4. Playwright MCP

**Что это:** MCP-сервер, экспортирующий операции Playwright как инструменты для клиента (например, Cursor).

**Плюсы:**

- Инструменты доступны **прямо в сессии чата** — меньше контекстных переключений «чат ↔ терминал».
- Точный контроль: навигация, селекторы, скриншоты — предсказуемо для скриптоподобных сценариев.
- Хорошо сочетается с политиками разрешений Cursor на вызов инструментов.

**Минусы:**

- Нужна установка и настройка конкретного MCP-сервера, переменные окружения, иногда отладка совместимости версий.
- Разные реализации серверов отличаются набором действий и стабильностью.
- Менее «одной фразой закрыть задачу», чем NL-обёртка Stagehand, если сервер низкоуровневый.

**Когда выбирать:** плотная итерация с UI из чата, когда важны явные шаги и повторяемость; когда в команде уже стандартизировали Playwright MCP.

---

## 5. Рекомендуемая стратегия для агента (как «лучше мне»)

1. **По умолчанию в этом проекте:** `vibe-tools browser` — уже интегрирован в правила, конфиг и сценарии «открой / сделай / извлеки».
2. **Если пользователь держит открытым Chrome с отладкой:** `--connect-to` и `url=current`, чтобы не терять состояние между шагами.
3. **Если нужен тесный цикл в чате без CLI:** подключить **Playwright MCP** и использовать его для пошаговой работы.
4. **Если цель — вынести автоматизацию в код/сервис на Python:** рассмотреть **browser-use** (или чистый Playwright в коде) отдельно от сессии Cursor.

---

## 6. Ограничения и честность

- Любой вариант требует рабочего браузера/Playwright и сети; на CI и песочницах возможны отдельные ограничения.
- ИИ не «видит» браузер сам по себе — только то, что вернули инструменты (скриншот, HTML, логи, текст ответа NL-движка).
- Стоимость и латентность зависят от выбранной модели (в т.ч. у Stagehand через vibe-tools).
